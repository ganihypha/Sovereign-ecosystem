## SECTION 6: 7 ANTI-PATTERNS WAJIB HAFAL

> ⚠️ Ini bukan opsional. Hafal sebelum exam day. Ini adalah "trap" yang dirancang untuk menjebak orang yang tahu konsepnya tapi belum paham production implications-nya.

### Anti-Pattern 1: Prompt-Based Ordering Enforcement
```
❌ SALAH: "Tambahkan few-shot examples untuk enforce urutan tool call"
✅ BENAR: "Gunakan programmatic prerequisites — ordering adalah 
          compliance constraint, bukan style preference"

WHY: AI bisa ignore prompt tapi tidak bisa ignore code. 
     Kalau urutan WAJIB → paksa di code, bukan di prompt.
```

### Anti-Pattern 2: Self-Reported Confidence for Escalation
```
❌ SALAH: "Route ke human ketika model melaporkan low confidence"
✅ BENAR: "Gunakan deterministic threshold logic — LLM confidence 
          tidak terkalibrasi dengan baik, terutama pada kasus tersulit"

WHY: Model paling sering "salah percaya diri" justru di kasus edge-case 
     yang paling kritis. Threshold deterministik jauh lebih reliable.
```

### Anti-Pattern 3: Batch API for Blocking Workflows
```
❌ SALAH: "Route semua workflows ke Batch API untuk hemat biaya"
✅ BENAR: "Batch tidak ada SLA — blocking user-facing workflows 
          tetap butuh real-time API, apapun biayanya"

WHY: Batch API tidak ada guaranteed response time. 
     User-facing = real-time. Internal heavy tasks = Batch OK.
```

### Anti-Pattern 4: Larger Context Window = Better Attention
```
❌ SALAH: "Tingkatkan context window untuk kurangi attention dilution"
✅ BENAR: "Split ke focused per-file passes — window size tidak 
          menyelesaikan masalah distribusi attention"

WHY: Makin panjang context → attention makin tersebar. 
     Solusi: pisahkan task, bukan perbesar window.
```

### Anti-Pattern 5: Silent Failure on Subagent Error
```
❌ SALAH: "Return empty results ketika subagent gagal"
✅ BENAR: "Return structured error context — silent failures 
          mencegah coordinator agent untuk recover dengan benar"

WHY: Orchestrator perlu tau APA yang gagal dan KENAPA 
     untuk bisa ambil keputusan retry/fallback yang tepat.
```

### Anti-Pattern 6: All Tools to All Agents
```
❌ SALAH: "Beri setiap agent akses ke full tool set (18 tools sekaligus)"
✅ BENAR: "Scope tools ke setiap agent role — 4-5 focused tools 
          jauh lebih reliable daripada 18 tools sekaligus"

WHY: Terlalu banyak tools = model bingung memilih.
     Sovereign Engine: ScoutScorer hanya butuh scraper + supabase.read.
     Bukan akses ke wa.send atau order.create!
```

### Anti-Pattern 7: Prompt-Only Output Enforcement
```
❌ SALAH: "Instruksikan model di prompt untuk selalu return valid JSON"
✅ BENAR: "Gunakan JSON schema validation + retry loop — 
          prompt saja tidak bisa guarantee output contract compliance"

WHY: Model bisa saja return "I can't format this as JSON" 
     atau format yang salah. Validasi programmatik + retry = satu-satunya guarantee.
```

### Mental Model Paling Penting untuk CCA-F
```
RULE: "Kalau behavior WAJIB dijamin → paksa lewat CODE, bukan PROMPT"

Diagram keputusan:
Is behavior GUARANTEED needed?
  YES → Programmatic enforcement (code/schema/threshold)
  NO  → Prompt-based guidance acceptable

Examples:
  Ordering of tool calls    → Code (programmatic)
  JSON output format        → Schema validation (programmatic)
  Escalation threshold      → Deterministic threshold (programmatic)
  Message tone/style        → Prompt (not guaranteed = OK)
  Content creativity        → Prompt (not guaranteed = OK)
  Summarization quality     → Prompt (not guaranteed = OK)
```
