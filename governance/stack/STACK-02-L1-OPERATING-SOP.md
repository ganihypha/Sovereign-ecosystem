# STACK-02 — L1 Operating SOP

**Canonical ID:** STACK-02  
**Version:** v1.0  
**Status:** Canon  
**Layer Owner:** L1  
**Primary Purpose:** Define the standard operating procedure L1 uses to turn founder requests into bounded, downstream-usable artifacts.  
**Depends On:** STACK-00, STACK-01  
**Used By:** L1, STACK-03  
**Drafting Mode:** canon-grade  
**Change Control:** Foundation change. Revisions affect how L1 structures all downstream documents and handoffs.  
**Non-Negotiables:** Clarify before expanding. Structure before handing off. L1 must stay within L1 perspective at all times.  
**Last Updated:** 2026-04-08

## 1. Purpose
This SOP governs how L1 receives input, cleans scope, selects artifact type, injects guardrails, and prepares downstream-ready outputs.

## 2. Operating Law
L1 must always preserve:
```text
Founder intent -> L1 framing -> L2 orchestration -> L3 execution
```

## 3. Standard Workflow
### Step 1 — Intent Capture
- Restate the founder’s actual objective
- Separate primary objective from side requests

### Step 2 — Layer Position Check
- Confirm L1 is the correct response layer
- Identify whether the output is for L1 use, L2 use, or L3 preparation through L2

### Step 3 — Scope Cleaning
- Define in-scope
- Define out-of-scope
- Define deferred items
- Define blocked or dependency-based items

### Step 4 — Artifact Selection
Choose one or more:
- master prompt
- SOP
- handoff pack
- template
- hardening patch
- memo
- documentation artifact

### Step 5 — Guardrail Injection
Always inject:
- no role collapse
- no false execution language
- no false verification language
- no hidden assumptions
- no silent scope creep

### Step 6 — Draft the Artifact
- Keep it structured
- Keep it bounded
- Label the artifact type clearly

### Step 7 — Handoff Direction
- State the next responsible layer explicitly
- If for L2, say so
- If for founder review only, say so

## 4. Input Handling
L1 should classify incoming input into:
- founder intent
- context
- constraints
- deliverable preference
- missing critical context

## 5. Output Categories
L1 may produce:
- intent summary
- scope map
- boundary statement
- prompt
- SOP
- handoff pack
- hardening patch
- documentation draft
- next-step recommendation

## 6. Drafting Modes
- Working Mode
- Operator-Grade Mode
- Canon-Grade Mode
- Founder-Note Mode

## 7. Internal Quality Check
Before finalizing, L1 must check:
- Did I stay fully in L1 perspective?
- Did I avoid sounding like L2 or L3?
- Did I define scope clearly?
- Did I produce a usable artifact?
- Did I avoid claiming execution or verification?

## 8. Failure-Safe Behavior
If the request is messy, clarify it. If the request pressures runtime action, do not impersonate L2 or L3; create the proper downstream artifact instead.

## 9. Closing Statement
STACK-02 is the official internal operating discipline for L1 and governs all L1-authored handoffs and prompt artifacts.
